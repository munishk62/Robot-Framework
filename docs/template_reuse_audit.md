# Template Reuse Audit (December 2025)

## Approach
- Parsed every base template in [test_data/entities](test_data/entities) and compared each variant against the entity "default" using key-set and value diffs.
- Applied a reuse heuristic: if two templates differ in <=40% of their attributes or only vary by value literals, treat them as override candidates unless they introduce new structural fields.
- Manually reviewed entities whose variants add nested collections or distinct workflow flags to confirm whether a dedicated template is still warranted.

## Current Footprint
- 32 entities provide base templates, totalling 107 named templates.
- Seven entities (day_off, availability, schedule_generation_setup, store, shift, time_off, timecard_punch) account for 53% of the templates and most duplication pressure.

## Automation Concept for Cleanup
- Generate a machine-readable report (JSON/CSV) from the comparison script capturing diff ratios, differing keys, and suggested override fields per template.
- Build a CLI utility that, for each entity flagged above, rewrites `base_templates.json` to keep only the recommended templates and exports a mapping of removed template names to override payloads.
- Emit companion migration hints (entity, removed template, override dict) so Robot tests can be auto-patched by replacing `template_name` arguments with inline overrides.
- Add a `--dry-run` mode that only prints the prospective removals to aid review.

## Guardrails to Prevent Future Proliferation
- Extend pre-commit (or Robocop) with a hook that rejects new templates when an entity already has a recommended override-able template; leverage the diff-ratio calculation to flag likely duplicates during PR review.
- Update [docs/SIMPLIFIED_DATA_PROVIDER_GUIDE.md](docs/SIMPLIFIED_DATA_PROVIDER_GUIDE.md) with explicit examples that pass overrides instead of copying templates, and reference this audit file.
- Enhance `TestDataLibrary` keywords to expose convenience overrides (e.g., `Get Day Off Data  template_name=default  reason=${reason}`) reducing the impulse to add new templates.
- Require reviewers to validate that proposed templates introduce new structural fields (arrays/maps) before approving PRs; provide a checklist linking back to this audit.
