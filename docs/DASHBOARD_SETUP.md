# Dashboard Setup and Architecture

## Overview

The WFM Test Automation Dashboard uses a multi-branch architecture to keep metrics data isolated and avoid conflicts with main development.

### Branch Strategy

- **`main`**: Main development branch (protected). Contains dashboard UI code (`docs/dashboard/`)
- **`dashboard-data`**: Orphan branch containing only metrics data (`docs/dashboard/data/`)
- **`gh-pages`**: Automatically generated by GitHub Actions. Serves published dashboard

### Data Flow

```
Jenkins tests on main
    ↓
output.xml generated
    ↓
Jenkins: Parse → JSON metrics
    ↓
Commit JSON to dashboard-data branch (orphan)
    ↓
GitHub Actions triggered on push to dashboard-data
    ↓
Checkout main (UI) + Fetch data from dashboard-data (temp dir)
    ↓
Overlay data → docs/dashboard/data/
    ↓
Publish entire docs/dashboard/ to gh-pages
    ↓
 Live dashboard served
```

### 1. Verify Jenkins Configuration

Jenkins pipeline should use the `dashboard-data` branch (default). When running a test with dashboard enabled:

```groovy
@Library('wfm-qe-libs') _
wfmQAPuneJenkins([
    "testEnv": "QA29_B0",
    "testPath": "tests/web/",
    "agentLabel": "SWS_UBU_02",
    "dashboardUpdateEnabled": true,  // Enables dashboard publishing
    "dashboardBranch": "dashboard-data"  // Optional: explicitly set (default)
])
```

## Workflow: Running Tests with Dashboard Updates

1. **Jenkins runs tests** on `main` branch with `PUBLISH_DASHBOARD=true`
2. **Tests complete**, `output.xml` is generated
3. **Jenkins post-build stage** (`updateDashboardData`):
   - Creates temp workspace
   - Clones `dashboard-data` branch (shallow, single branch)
   - Runs `dashboard_results_updater.py` to generate JSON metrics
   - Commits and pushes JSON files to `dashboard-data` branch
4. **GitHub Actions** (`sync-dashboard-data.yml`):
   - Detects push to `dashboard-data`
   - Extracts JSON files
   - Merges them into `gh-pages` branch
5. **GitHub Pages** automatically serves updated dashboard

## Troubleshooting

### Dashboard data not updating
1. Check Jenkins logs: `PUBLISH_DASHBOARD` parameter is `true`
2. Verify `dashboard-data` branch exists: `git ls-remote origin dashboard-data`
3. Verify GitHub token credentials in Jenkins: Check `kg_pat_pipeline` has repo access
4. Check GitHub Actions: `.github/workflows/sync-dashboard-data.yml` runs successfully

### Want to reset dashboard data
```bash
git checkout dashboard-data
git reset --hard origin/main  # This won't work—these are orphan branches
# Instead, just recreate the orphan branch from scratch (see setup above)
```

### How to manually test the updater
```bash
# Generate sample output.xml
python executor.py tests/web/ --test-env QA29_B0 --dry-run

# Run the updater
python dev_utils/dashboard_results_updater.py \
  --output-xml results_QA29_B0/output.xml \
  --env-name QA29_B0 \
  --data-dir docs/dashboard/data
```

## Notes

- Dashboard data is independent of main branch—no conflicts
- Orphan branch keeps history clean
- JSON updates are atomic (only data files committed)
- GitHub Pages serves dashboard automatically
- All three data files are required by the frontend
